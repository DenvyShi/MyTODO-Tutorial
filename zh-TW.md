# 我如何從零開始打造自己的待辦事項 App（你也可以！）

> 一份適合新手的全端任務管理 Web 應用開發指南 —— 不需要資工學位也能看懂。

## 🤔 為什麼要自己動手做？

你可能用過 Microsoft To Do、Todoist 或 Apple 提醒事項。它們都很好用 —— 但如果你能自己做一個呢？不只是為了好玩，而是藉此**真正理解現代 Web 應用程式的運作原理**。

這正是我做的事。在這篇文章中，我會帶你走一遍我如何從零打造 **MyTODO** 的完整過程，並用最淺顯的方式解釋每個概念。

---

## 📸 它長什麼樣子？

MyTODO 看起來和用起來都像一個真正的生產力工具：

- 📋 **多清單管理**（工作、個人、購物等）—— 就像 Microsoft To Do
- ✅ **完整任務管理** —— 新增、編輯、完成、刪除任務
- 🎯 **優先級設定** —— 高、中、低
- 📅 **截止日期與時間**
- 🖱️ **拖拽操作** —— 用滑鼠拖動任務到不同清單
- 📱 **手機友好** —— 在任何螢幕尺寸上都能正常使用
- 🔔 **智慧提醒** —— 任務到期前透過 Telegram 或 Email 通知你
- 🔐 **使用者帳號** —— 帳號密碼登入，資料完全私密

## 🧱 建構基石（別擔心，我會一一解釋）

在開始之前，讓我先解釋關鍵技術。把開發 Web 應用想像成蓋房子：

### 前端（Frontend）—— 「門面」（使用者看到的東西）

| 技術 | 它做什麼 | 可以想成... |
|---|---|---|
| **React** | 建構互動式 UI 元件 | 牆壁、窗戶和門 |
| **Vite** | 快速開發工具 | 讓施工更快的電動工具 |
| **Tailwind CSS** | 不用寫太多 CSS 就能讓介面變好看 | 盒裝室內設計方案 |
| **@dnd-kit** | 拖放功能 | 可以重新排列的家具 |

### 後端（Backend）—— 「引擎室」（幕後運作）

| 技術 | 它做什麼 | 可以想成... |
|---|---|---|
| **Node.js** | 在伺服器上執行 JavaScript | 驅動一切的電力 |
| **Express** | 處理網路請求 | 把請求路由到正確地方的櫃台人員 |
| **SQLite** | 把資料存在一個檔案裡 | 一個檔案櫃 |
| **JWT** | 安全的使用者認證 | 活動上的 VIP 手環 |

### 部署 —— 把應用放上網際網路

| 技術 | 它做什麼 |
|---|---|
| **Linux (WSL2)** | 伺服器環境 |
| **Systemd** | 讓應用 24 小時運行 |
| **Cloudflare Tunnel** | 不用開放路由器端口就能獲得公開 URL |
| **HTTPS** | 加密使用者與伺服器之間的資料傳輸 |

## 🏗️ 應用如何運作（架構簡化版）

```
┌─────────────┐         ┌─────────────┐         ┌──────────┐
│   瀏覽器     │ ──WiFi──▶│   伺服器     │ ──讀取──▶│  SQLite  │
│  （前端）    │◀──WiFi──│  （後端）    │◀─寫入──│  資料庫  │
└─────────────┘         └──────┬──────┘         └──────────┘
                               │
                    ┌──────────┼──────────┐
                    ▼                     ▼
              ┌──────────┐         ┌──────────┐
              │ Telegram  │         │  Email   │
              │  （機器人）│         │ （SMTP） │
              └──────────┘         └──────────┘
```

**當你打開應用時，發生了什麼事：**

1. 瀏覽器載入 React 前端（好看的介面）
2. 當你新增任務時，React 發送訊息給 Express 伺服器
3. 伺服器檢查你的 JWT token（你登入了嗎？）
4. 伺服器將任務儲存到 SQLite 資料庫
5. 任務到期時，伺服器透過 Telegram 或 Email 發送提醒

## 🔑 關鍵概念解釋

### 1. 客戶端-伺服器模型（Client-Server Model）
想像一下餐廳：
- **客戶端（瀏覽器）** = 你，看菜單並點餐
- **伺服器（後端）** = 廚房，準備你的食物
- **API** = 服務生，在你和廚房之間傳遞訂單

### 2. REST API
「服務生」使用一種標準語言叫做 REST：
- `GET /api/tasks` → 「給我看所有任務」
- `POST /api/tasks` → 「新增一個任務」
- `PUT /api/tasks/123` → 「更新第 123 號任務」
- `DELETE /api/tasks/123` → 「刪除第 123 號任務」

### 3. JWT 認證
登入時，伺服器給你一個 **token**（就像夜店的 VIP 手環）。之後每次發請求，你都要出示這個 token。伺服器驗證後說：「OK，你可以進來。」

### 4. 資料庫（SQLite）
跟大型資料庫（MySQL、PostgreSQL）不同，SQLite 把所有東西存在**一個檔案**裡（`mytodo.db`）。對個人專案來說簡直完美 —— 不需要額外的資料庫伺服器！

## 🧪 測試：為什麼 94.69% 的覆蓋率很重要

我寫了 **83 個自動化測試**，幾乎涵蓋了每一行程式碼。為什麼？

- **信心保證**：如果我改了某個東西導致測試失敗，我能立刻發現
- **活的文件**：測試精確展示了程式碼應有的行為
- **專業實踐**：在真正的公司裡，測試是必須的

測試範例：
```javascript
test('應該能建立新任務', async () => {
  const response = await request(app)
    .post('/api/tasks')
    .set('Authorization', `Bearer ${token}`)
    .send({
      title: '買菜',
      list_id: listId,
      priority: 'high',
      due_date: '2026-04-15'
    });
  
  expect(response.status).toBe(201);
  expect(response.body.title).toBe('買菜');
});
```

## 🌐 我如何把它部署到網際網路上

### 問題
我的應用跑在家用伺服器上。如何讓任何人都能存取，又不用花錢租雲端主機？

### 解決方案：Cloudflare Tunnel
```
你的手機 ──▶ Cloudflare CDN ──▶ 隧道 ──▶ 你的家用伺服器
              (mytodo.first.pet)           （在 NAT/路由器後面）
```

Cloudflare Tunnel 從我的伺服器到 Cloudflare 的全球網路建立一條安全連線。不需要：
- 在路由器上開放端口（安全風險！）
- 購買固定 IP 地址
- 手動設定 SSL 憑證

### 步驟
1. 在伺服器上安裝 `cloudflared`
2. 建立隧道：`cloudflared tunnel create mytodo`
3. 設定指向 `localhost:3001`
4. 指定域名：`mytodo.first.pet`
5. 設定 systemd 服務讓它開機自動啟動
6. 完成！應用現在可以透過 HTTPS 從全球存取 🎉

## 💡 學到的經驗

### 做得好的地方
- **從簡單開始，逐步迭代**：我沒有試圖一次建好所有東西。先是基本的增刪改查，然後是拖拽功能，再來是通知系統。
- **邊寫邊測試**：跟著程式碼一起寫測試，省下了無數的除錯時間
- **SQLite 被低估了**：對個人專案來說，它就夠用了

### 如果重來會做得不同的地方
- **更早規劃資料庫結構**：新增功能時我不得不遷移資料
- **手機優先設計**：我先做了桌面版，後來才修手機版
- **錯誤處理**：我應該從一開始就思考「什麼地方會出錯」
