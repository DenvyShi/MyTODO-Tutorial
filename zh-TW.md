## 🎯 這篇文章的真正重點

這**不是**一篇教你做待辦事項 App 的教程。那種教程網路上有上千篇。

這是一篇關於**如何用 AI 來開發軟體**的教程——即使你不會寫程式。這個待辦事項應用（我叫它 MyTODO）只是載體。我想分享的是**過程**：如何拆解想法、跟 AI 溝通、迭代結果，最終產出達到正式環境品質的軟體。

讀完這篇文章，你會了解：
- 如何組織你的請求，讓 AI 產出好程式碼
- 如何分階段建構複雜軟體（而不是一次全做）
- 如何測試、除錯和部署——全部透過對話完成
- 為什麼「寫程式用的 Prompt 工程」是一門真正的、可學的技能

**劇透：這個應用最終達到了 94.69% 的測試覆蓋率、83 個自動化測試，並且在我的家用伺服器上 24 小時運行——全球透過 HTTPS 存取。而我從未打開過程式碼編輯器。**

## 🧠 心態轉換：你是產品經理，不是程式設計師

當你用 AI 建構軟體時，你的角色改變了。你不再是寫程式碼的人，你變成了**產品經理**——負責：

1. **定義要做什麼**（功能、使用者體驗、限制條件）
2. **把工作拆成階段**（不要一次要求所有東西）
3. **審查產出**（能用嗎？看起來對嗎？）
4. **持續迭代**（修 bug、加功能、打磨）

AI 是你的開發團隊。它寫程式碼、跑測試、安裝套件、做部署。但它需要明確的方向。

這樣想：如果你能把一個想法解釋得清楚到人類工程師能建構出來，那你就能解釋得清楚到讓 AI 建構出來。

## 💬 完整的 Prompt 序列

以下是我用來建構 MyTODO 的實際 Prompt，按階段組織。每條 Prompt 都建立在前一條的基礎上。**這是這篇文章最重要的部分**——研究它的結構，不只是內容。

### 階段一：打好地基

**Prompt 1 — 定義願景：**
> 我想做一個類似 Microsoft To Do 的待辦事項 Web 應用。技術棧：
> - 後端：Node.js + Express + SQLite + JWT 認證
> - 前端：React + Vite + Tailwind CSS
> - 通知：Telegram Bot + Email（Nodemailer）
> 
> 需要的功能：
> - 使用者註冊和登入
> - 多個任務清單（如工作、個人、購物）
> - 任務增刪改查，含標題、描述、優先級（高/中/低）、截止日期時間
> - 拖拽排序
> - Telegram 和 Email 到期提醒
> 
> 先建立專案結構並實作後端。

**為什麼這樣寫有效：**
- ✅ 明確的技術棧（不是「做一個 Web App」）
- ✅ 清楚的功能列表
- ✅ 範圍限定在後端（先做重要的事）
- ✅ 提供了參考對象（「像 Microsoft To Do」）

**Prompt 2 — 建構使用者介面：**
> 現在建立 React 前端：
> - 現代風格的登入/註冊頁面
> - 側邊欄顯示所有清單的儀表板
> - 可在清單之間拖拽的任務卡片
> - 新增/編輯任務的彈窗，包含所有欄位
> - 使用 Tailwind CSS 樣式，預設深色主題

**Prompt 3 — 串接所有東西：**
> 建立前端的 API 層連接後端：
> - 認證 API（登入、註冊、token 管理）
> - 清單 API（增刪改查）
> - 任務 API（增刪改查）
> - 在 localStorage 處理 JWT token
> - 錯誤處理和載入狀態

---

### 階段二：從「能用」到「好用」

**Prompt 4 — UI 打磨：**
> 加入這些 UI 改進：
> - 側邊欄可折疊（切換按鈕）
> - 每個清單分配隨機彩色圖標（30 種不重複）
> - 新增「所有任務」視圖
> - 新增「今天」篩選器
> - 新增「已完成」區域
> - 拖拽任務到不同清單時，應該更改其 list_id

**為什麼這個階段重要：** 第一階段讓你得到一個能用的產品。這個階段讓它變成一個**好用的**產品。永遠不要在第一階段就試圖打磨——先讓它能跑。

**Prompt 5 — 多語言支援：**
> 為應用加入國際化（i18n）：
> - 支援英文和繁體中文（zh-TW）
> - 在設定中加入語言切換
> - 為所有 UI 文字建立語言檔

---

### 階段三：品質保證

**Prompt 6 — 自動化測試：**
> 為後端撰寫全面的測試：
> - 測試所有認證路由（註冊、登入、無效憑證）
> - 測試清單和任務的所有增刪改查操作
> - 測試授權（使用者只能存取自己的資料）
> - 測試邊界情況（空白欄位、重複名稱、無效 ID）
> - 使用 Jest + Supertest
> - 目標覆蓋率 90% 以上

**結果：** 83 個測試，94.69% 覆蓋率。每一個測試都是 AI 寫的。我只是提了這個要求。

**為什麼測試在使用 AI 編程時特別重要：** AI 生成的程式碼通常在正常流程下能運作。但邊界情況——空輸入、未授權存取、重複資料——才是 bug 藏身之處。要求 AI 寫測試，迫使它思考這些情況，也讓你對程式碼品質有信心。

---

### 階段四：上線！

**Prompt 7 — 部署自動化：**
> 建立部署腳本：
> - 安裝所有依賴（前後端都執行 npm install）
> - 建置 React 前端用於正式環境
> - 建立 systemd 服務檔，讓應用開機自動啟動
> - 伺服器在正式環境模式下提供建置好的前端
> - 使用 port 3001

**Prompt 8 — 公開存取：**
> 幫我設定 Cloudflare Tunnel 讓應用公開存取：
> - 我在 Cloudflare 上有域名 `first.pet`
> - 我想要子域名 `mytodo.first.pet`
> - 設定命名隧道（不是快速隧道）
> - 為隧道建立 systemd 服務讓它自動啟動

**部署架構：**
```
任何裝置 ──▶ Cloudflare CDN (HTTPS) ──▶ 隧道 ──▶ 我的家用伺服器
              mytodo.first.pet                      (port 3001)
```

---

### 階段五：根據實際使用迭代

**Prompt 9 — 修 bug：**
> 修復這些問題：
> - 拖拽任務到另一個清單後，應該立即從舊清單消失（不要等到重新整理）
> - 每個清單的任務計數應該即時更新
> - 刪除清單前加入確認對話框

**這才是真正的軟體開發方式。** 你建構、使用、發現問題、修復問題。AI 讓這個循環變得極快。

## 🔑 AI 輔助編程的 7 條原則

透過用這種方式建構多個專案，我學到了這些：

### 1. 明確指定技術棧
❌ 「做一個 Web App」
✅ 「使用 React + Express + SQLite，搭配 JWT 認證」

你不指定的話，AI 會隨機選擇。隨機選擇會導致不相容的技術組合。

### 2. 用項目符號列出功能
❌ 「要有正常的功能」
✅ 「功能：使用者登入、多清單、拖拽、截止日期、優先級」

### 3. 分階段建構（這是最關鍵的！）
新手最大的錯誤：在一條 Prompt 裡要求所有東西。AI 會試著全部做，結果是一團亂。

**正確的做法：**
1. 地基（後端 + 基本前端）
2. 功能（拖拽、通知）
3. 打磨（UI 改進、動畫）
4. 品質（測試、錯誤處理）
5. 部署（腳本、隧道、服務）
6. 迭代（修 bug、新功能）

### 4. 提供參考對象
「我想要一個像 Microsoft To Do 的東西」立刻給 AI 一個關於 UI、功能和使用者體驗的腦海模型。

### 5. 審查並迭代
每個階段完成後，看看結果。能用嗎？看起來對嗎？然後要求修改。這很正常——即使人類工程師也要迭代很多版本。

### 6. 及早要求測試
測試在你部署前就能抓住 bug。它們也是程式碼應有行為的活文件。

### 7. 逐步部署
不要等到所有東西都完美才部署。盡早上線，自己使用，遇到問題再修。
