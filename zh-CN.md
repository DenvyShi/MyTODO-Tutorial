## 🧠 心态转换：你是产品经理，不是程序员

当你用 AI 建构软件时，你的角色改变了。你不再是写代码的人，你变成了**产品经理**——负责：

1. **定义要做什么**（功能、用户体验、限制条件）
2. **把工作拆成阶段**（不要一次要求所有东西）
3. **审查产出**（能用吗？看起来对吗？）
4. **持续迭代**（修 bug、加功能、打磨）

AI 是你的开发团队。它写代码、跑测试、安装套件、做部署。但它需要明确的方向。

这样想：如果你能把一个想法解释得清楚到人类工程师能建构出来，那你就能解释得清楚到让 AI 建构出来。

## 💬 完整的 Prompt 序列

以下是我用来建构 MyTODO 的实际 Prompt，按阶段组织。每条 Prompt 都建立在前一条的基础上。**这是这篇文章最重要的部分**——研究它的结构，不只是内容。

### 阶段一：打好地基

**Prompt 1 — 定义愿景：**
> 我想做一个类似 Microsoft To Do 的待办事项 Web 应用。技术栈：
> - 后端：Node.js + Express + SQLite + JWT 认证
> - 前端：React + Vite + Tailwind CSS
> - 通知：Telegram Bot + Email（Nodemailer）
> 
> 需要的功能：
> - 用户注册和登录
> - 多个任务清单（如工作、个人、购物）
> - 任务增删改查，含标题、描述、优先级（高/中/低）、截止日期时间
> - 拖拽排序
> - Telegram 和 Email 到期提醒
> 
> 先建立项目结构并实现后端。

**为什么这样写有效：**
- ✅ 明确的技术栈（不是「做一个 Web App」）
- ✅ 清楚的功能列表
- ✅ 范围限定在后端（先做重要的事）
- ✅ 提供了参考对象（「像 Microsoft To Do」）

**Prompt 2 — 建构使用者介面：**
> 现在建立 React 前端：
> - 现代风格的登录/注册页面
> - 侧边栏显示所有清单的仪表板
> - 可在清单之间拖拽的任务卡片
> - 新增/编辑任务的弹窗，包含所有字段
> - 使用 Tailwind CSS 样式，默认深色主题

**Prompt 3 — 串联所有东西：**
> 建立前端的 API 层连接后端：
> - 认证 API（登录、注册、token 管理）
> - 清单 API（增删改查）
> - 任务 API（增删改查）
> - 在 localStorage 处理 JWT token
> - 错误处理和加载状态

---

### 阶段二：从「能用」到「好用」

**Prompt 4 — UI 打磨：**
> 加入这些 UI 改进：
> - 侧边栏可折叠（切换按钮）
> - 每个清单分配随机彩色图标（30 种不重复）
> - 新增「所有任务」视图
> - 新增「今天」筛选器
> - 新增「已完成」区域
> - 拖拽任务到不同清单时，应该更改其 list_id

**为什么这个阶段重要：** 第一阶段让你得到一个能用的产品。这个阶段让它变成一个**好用的**产品。永远不要在第一阶段就试图打磨——先让它能跑。

**Prompt 5 — 多语言支持：**
> 为应用加入国际化（i18n）：
> - 支持英文和繁体中文（zh-TW）
> - 在设置中加入语言切换
> - 为所有 UI 文字建立语言文件

---

### 阶段三：品质保证

**Prompt 6 — 自动化测试：**
> 为后端撰写全面的测试：
> - 测试所有认证路由（注册、登录、无效凭证）
> - 测试清单和任务的所有增删改查操作
> - 测试授权（用户只能存取自己的数据）
> - 测试边界情况（空白字段、重复名称、无效 ID）
> - 使用 Jest + Supertest
> - 目标覆盖率 90% 以上

**结果：** 83 个测试，94.69% 覆盖率。每一个测试都是 AI 写的。我只是提了这个要求。

**为什么测试在使用 AI 编程时特别重要：** AI 生成的代码通常在正常流程下能运作。但边界情况——空输入、未授权存取、重复数据——才是 bug 藏身之处。要求 AI 写测试，迫使它思考这些情况，也让你对代码品质有信心。

---

### 阶段四：上线！

**Prompt 7 — 部署自动化：**
> 建立部署脚本：
> - 安装所有依赖（前后端都执行 npm install）
> - 建置 React 前端用于正式环境
> - 建立 systemd 服务文件，让应用开机自动启动
> - 服务器在正式环境模式下提供建置好的前端
> - 使用 port 3001

**Prompt 8 — 公开存取：**
> 帮我设定 Cloudflare Tunnel 让应用公开存取：
> - 我在 Cloudflare 上有域名 `first.pet`
> - 我想要子域名 `mytodo-tutorial.first.pet`
> - 设定命名隧道（不是快速隧道）
> - 为隧道建立 systemd 服务让它自动启动

**部署架构：**
```
任何装置 ──▶ Cloudflare CDN (HTTPS) ──▶ 隧道 ──▶ 我的家用服务器
              mytodo-tutorial.first.pet                      (port 3002)
```

---

### 阶段五：根据实际使用迭代

**Prompt 9 — 修 bug：**
> 修复这些问题：
> - 拖拽任务到另一个清单后，应该立即从旧清单消失（不要等到刷新）
> - 每个清单的任务计数应该实时更新
> - 删除清单前加入确认对话框

**这才是真正的软件开发方式。** 你建构、使用、发现问题、修复问题。AI 让这个循环变得极快。

## 🔑 AI 辅助编程的 7 条原则

透过用这种方式建构多个项目，我学到了这些：

### 1. 明确指定技术栈
❌ 「做一个 Web App」
✅ 「使用 React + Express + SQLite，搭配 JWT 认证」

你不指定的话，AI 会随机选择。随机选择会导致不相容的技术组合。

### 2. 用项目符号列出功能
❌ 「要有正常的功能」
✅ 「功能：用户登录、多清单、拖拽、截止日期、优先级」

### 3. 分阶段建构（这是最关键的！）
新手最大的错误：在一条 Prompt 里要求所有东西。AI 会试着全部做，结果是一团乱。

**正确的做法：**
1. 地基（后端 + 基本前端）
2. 功能（拖拽、通知）
3. 打磨（UI 改进、动画）
4. 品质（测试、错误处理）
5. 部署（脚本、隧道、服务）
6. 迭代（修 bug、新功能）

### 4. 提供参考对象
「我想要一个像 Microsoft To Do 的东西」立刻给 AI 一个关于 UI、功能和用户体验的脑海模型。

### 5. 审查并迭代
每个阶段完成后，看看结果。能用吗？看起来对吗？然后要求修改。这很正常——即使人类工程师也要迭代很多版本。

### 6. 及早要求测试
测试在你部署前就能抓住 bug。它们也是代码应有行为的活文件。

### 7. 逐步部署
不要等到所有东西都完美才部署。尽早上线，自己使用，遇到问题再修。
