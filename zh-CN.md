# 我如何从零开始打造自己的待办事项 App（你也可以！）

> 一份适合新手的全栈任务管理 Web 应用开发指南 —— 不需要计算机学位也能看懂。

## 🤔 为什么要自己动手做？

你可能用过 Microsoft To Do、Todoist 或 Apple 提醒事项。它们都很好用 —— 但如果你能自己做一个呢？不只是为了好玩，而是借此**真正理解现代 Web 应用程序的工作原理**。

这正是我做的事。在这篇文章中，我会带你走一遍我如何从零打造 **MyTODO** 的完整过程，并用最浅显的方式解释每个概念。

## 📸 它长什么样子？

MyTODO 看起来和用起来都像一个真正的生产力工具：

- 📋 **多清单管理**（工作、个人、购物等）—— 就像 Microsoft To Do
- ✅ **完整任务管理** —— 创建、编辑、完成、删除任务
- 🎯 **优先级设置** —— 高、中、低
- 📅 **截止日期与时间**
- 🖱️ **拖拽操作** —— 用鼠标拖动任务到不同清单
- 📱 **手机友好** —— 在任何屏幕尺寸上都能正常使用
- 🔔 **智能提醒** —— 任务到期前通过 Telegram 或 Email 通知你
- 🔐 **用户账号** —— 账号密码登录，数据完全私密

## 🧱 构建基石（别担心，我会一一解释）

在开始之前，让我先解释关键技术。把开发 Web 应用想象成盖房子：

### 前端（Frontend）—— "门面"（用户看到的东西）

| 技术 | 它做什么 | 可以想象成... |
|---|---|---|
| **React** | 构建交互式 UI 组件 | 墙壁、窗户和门 |
| **Vite** | 快速开发工具 | 让施工更快的电动工具 |
| **Tailwind CSS** | 不用写太多 CSS 就能让界面变好看 | 盒装室内设计方案 |
| **@dnd-kit** | 拖放功能 | 可以重新排列的家具 |

### 后端（Backend）—— "引擎室"（幕后运作）

| 技术 | 它做什么 | 可以想象成... |
|---|---|---|
| **Node.js** | 在服务器上运行 JavaScript | 驱动一切的电力 |
| **Express** | 处理网络请求 | 把请求路由到正确地方的前台人员 |
| **SQLite** | 把数据存在一个文件里 | 一个文件柜 |
| **JWT** | 安全的用户认证 | 活动上的 VIP 手环 |

### 部署 —— 把应用放上互联网

| 技术 | 它做什么 |
|---|---|
| **Linux (WSL2)** | 服务器环境 |
| **Systemd** | 让应用 24 小时运行 |
| **Cloudflare Tunnel** | 不用开放路由器端口就能获得公开 URL |
| **HTTPS** | 加密用户与服务器之间的数据传输 |

## 🏗️ 应用如何运作（架构简化版）

```
┌─────────────┐         ┌─────────────┐         ┌──────────┐
│   浏览器     │ ──WiFi──▶│   服务器     │ ──读取──▶│  SQLite  │
│  （前端）    │◀──WiFi──│  （后端）    │◀─写入──│  数据库  │
└─────────────┘         └──────┬──────┘         └──────────┘
                               │
                    ┌──────────┼──────────┐
                    ▼                     ▼
              ┌──────────┐         ┌──────────┐
              │ Telegram  │         │  Email   │
              │  （机器人）│         │ （SMTP） │
              └──────────┘         └──────────┘
```

**当你打开应用时，发生了什么事：**

1. 浏览器加载 React 前端（好看的界面）
2. 当你创建任务时，React 发送消息给 Express 服务器
3. 服务器检查你的 JWT token（你登录了吗？）
4. 服务器将任务保存到 SQLite 数据库
5. 任务到期时，服务器通过 Telegram 或 Email 发送提醒

## 🔑 关键概念解释

### 1. 客户端-服务器模型（Client-Server Model）
想象一下餐厅：
- **客户端（浏览器）** = 你，看菜单并点餐
- **服务器（后端）** = 厨房，准备你的食物
- **API** = 服务员，在你和厨房之间传递订单

### 2. REST API
"服务员"使用一种标准语言叫做 REST：
- `GET /api/tasks` → "给我看所有任务"
- `POST /api/tasks` → "创建一个新任务"
- `PUT /api/tasks/123` → "更新第 123 号任务"
- `DELETE /api/tasks/123` → "删除第 123 号任务"

### 3. JWT 认证
登录时，服务器给你一个 **token**（就像夜店的 VIP 手环）。之后每次发请求，你都要出示这个 token。服务器验证后说："OK，你可以进来。"

### 4. 数据库（SQLite）
跟大型数据库（MySQL、PostgreSQL）不同，SQLite 把所有东西存在**一个文件**里（`mytodo.db`）。对个人项目来说简直完美 —— 不需要额外的数据库服务器！

## 🌐 我如何把它部署到互联网上

### 问题
我的应用跑在家用服务器上。如何让任何人都能访问，又不用花钱租云主机？

### 解决方案：Cloudflare Tunnel
```
你的手机 ──▶ Cloudflare CDN ──▶ 隧道 ──▶ 你的家用服务器
              (mytodo.first.pet)           （在 NAT/路由器后面）
```

Cloudflare Tunnel 从我的服务器到 Cloudflare 的全球网络建立一条安全连接。不需要：
- 在路由器上开放端口（安全风险！）
- 购买固定 IP 地址
- 手动配置 SSL 证书

### 步骤
1. 在服务器上安装 `cloudflared`
2. 创建隧道：`cloudflared tunnel create mytodo`
3. 配置指向 `localhost:3001`
4. 指定域名：`mytodo.first.pet`
5. 配置 systemd 服务让它开机自动启动
6. 完成！应用现在可以通过 HTTPS 从全球访问 🎉

## 💡 学到的经验

### 做得好的地方
- **从简单开始，逐步迭代**：我没有试图一次建好所有东西。先是基本的增删改查，然后是拖拽功能，再来是通知系统。
- **边写边测试**：跟着代码一起写测试，省下了无数的调试时间
- **SQLite 被低估了**：对个人项目来说，它就够了

### 如果重来会做得不同的地方
- **更早规划数据库结构**：添加功能时我不得不迁移数据
- **手机优先设计**：我先做了桌面版，后来才修手机版
- **错误处理**：我应该从一开始就思考"什么地方会出错"
